# AI 하네스 설계 가이드

AI 에이전트를 예측 가능하고, 정확하고, 안전하게 만드는 하네스 설계 방법.

> *참고: 무신사 테크블로그 "AI 스페셜리스트와 자동사냥 하네스로 제어하는 AI 파이프라인"*

---

## 핵심 철학: Agent = Model + Harness

LLM 단독으로는 범용 기계입니다 — 뭐든 할 수 있다는 건 잘못된 것도 할 수 있다는 뜻입니다. **하네스**는 모델의 동작을 특정 작업에 맞게 형성하는 제어 환경입니다.

> "이 프로젝트 코드의 99%는 Claude가 작성했다. 엔지니어의 역할은 하네스를 설계하는 것이다."

AI 기반 워크플로우에서 엔지니어의 역할:
- 하네스 설계 (시스템 프롬프트, 에이전트 구조, 단계 정의)
- 파티 구성 (동시성 제한, 공급자 선택, 최대 반복 횟수)
- 실패 후 안전장치 추가
- 최종 판단 (출력 승인/거부)

---

## 세 가지 하네스 유형

### 1. 타이트 하네스 — 스페셜리스트

**사용 시점:** 명확하게 정의된, 반복 가능한, 고위험 작업.

```
특성:
✓ 좁은 역할: "10년 경력 Spring Boot/JPA 전문가"
✓ 제한된 출력: unified diff만, JSON 스키마만 등
✓ 명시적 금지 행동 목록
✓ 결정론적 출력 검증
```

**트레이드오프:** 특정 작업에서 높은 정확도; 범위 밖에서는 무용.

**이 프로젝트의 예시 에이전트:** `security-auditor`, `database-expert`, `harness-designer`

---

### 2. 느슨한 하네스 — 제너럴리스트

**사용 시점:** 탐색적, 개방형, 대화형 작업.

```
특성:
✓ 넓고 유연한 역할
✓ 자유로운 출력
✓ 각 단계에서 사람의 판단 필요
```

**트레이드오프:** 유연하지만 예측 불가; 사람의 적극적 개입 필요.

**예시:** 인터랙티브 모드의 Claude Code 자체.

---

### 3. 적응형 하네스 — 오케스트레이터

**사용 시점:** 여러 스페셜리스트를 결합하는 다단계 워크플로우.

```
특성:
✓ 스페셜리스트 에이전트 조율
✓ 이전 단계 출력 기반 라우팅
✓ 에이전트 간 컨텍스트 격리 관리
✓ 단계별 실패 처리
```

**트레이드오프:** 복잡한 자동화에 가장 강력; 설계 복잡도 가장 높음.

**이 프로젝트의 예시 에이전트:** `orchestrator`, `pipeline-orchestrator`

## 컨텍스트 격리: Context Rot 방지

긴 파이프라인에서 축적된 컨텍스트는 "Context Rot"을 유발합니다 — 초기 오류가 후속 단계를 오염시킵니다.

### 원칙
각 에이전트는 **필요한 컨텍스트만** 받습니다 — 그 이상은 안 됩니다.

```
❌ 나쁜 예: 모든 에이전트에 전체 대화 히스토리 전달
✅ 좋은 예: 이전 단계의 검증된 출력만 전달

❌ 나쁜 예: 한 에이전트가 모든 것을 읽고 모든 것을 함
✅ 좋은 예: 4개의 병렬 에이전트가 각각 하나씩 읽고 하나씩 보고
```

### 패턴: 병렬 정찰
```
2단계 — 병렬, 컨텍스트 격리:
  에이전트 A: 코드 구조 분석      ← 독립된 컨텍스트 윈도우
  에이전트 B: DB 메타데이터 수집  ← 독립된 컨텍스트 윈도우
  에이전트 C: 설정 확인           ← 독립된 컨텍스트 윈도우
  에이전트 D: 트레이드오프 분석   ← 독립된 컨텍스트 윈도우

3단계 — 합성:
  위의 4개 보고서만 받음 (전체 세션 히스토리 X)
```

## 타이트 하네스 구축: 단계별

### 1. 문제를 정확히 정의
```
단일하고 반복 가능한 작업이 무엇인가?
완벽한 출력은 어떤 모습인가? (이것이 성공 기준)
실패 모드는 무엇인가?
사람의 감독이 필요한 부분은 어디인가?
```

### 2. 스페셜리스트 페르소나 작성
```
당신은 [특정 도메인]의 [N년 경력] 전문가입니다.
당신은 [특정 작업]만 합니다. 그 외에는 없습니다.
```

### 3. 입력/출력 계약 정의
```
## 입력 계약
[받는 것의 정확한 형식]

## 출력 계약
다음 형식으로만 출력 — 그 외 없음:
[정확한 형식: unified diff / JSON / 구조화된 마크다운]
```

### 4. 금지 행동 목록
```
## 금지 사항
- [범위] 밖의 파일 절대 수정 금지
- 요청받지 않은 기능 추가 금지
- 정의된 형식 외의 출력 금지
```

### 5. 검증 레이어 추가
```
정적:  구문 검사, 스키마 적합성, 형식 일치
동적:  드라이런, EXPLAIN, 테스트 실행
판정: PASS | WARNING | MISMATCH | ERROR

기본값: 경고 (차단하지 않음), 비가역적 작업이라면 차단
```

### 6. 사람의 승인 지점 정의
```
AI가 자율적으로 결정하는 것은?   → 하네스 안에서
사람의 승인이 필요한 것은?       → 적용 전에
절대 자동화하지 않는 것은?      → 최종 머지, 프로덕션 배포
```

## 리뷰 루프 패턴

리뷰 루프는 여러 리뷰 에이전트를 병렬로 실행하고, 발견 사항을 중복 제거하며, 수정을 적용하고, 반복합니다.

```
라운드 N:
  리뷰 에이전트 병렬 실행:
    에이전트 1: 정확성 리뷰 (격리된 컨텍스트)
    에이전트 2: 보안 리뷰    (격리된 컨텍스트)
    에이전트 3: 성능 리뷰    (격리된 컨텍스트)
    에이전트 4: 스타일 리뷰  (격리된 컨텍스트)
  
  발견 사항 병합 → 중복 제거 → 수정 → 재리뷰

조기 종료 조건:
  - 모든 발견 사항 해결
  - 이전 라운드 대비 개선 없음
  - 최대 반복 횟수 도달
```

**다중 공급자 변형:** 동일한 4가지 관점을 2개의 AI 공급자로 실행 → 최대 8개 병렬 리뷰 → 중복 발견 사항 자동 병합.

## 스킬 시스템: 파일로 관리하는 프롬프트

프롬프트를 버전 관리하고 조합 가능하게 유지:

```
skills/
  analyst/
    system.md     ← 분석가 스페셜리스트 시스템 프롬프트
    user.md       ← 사용자 프롬프트 템플릿
  reviewer/
    system.md
    user.md
  shared/
    checklist.md  ← 여러 스킬에서 공유하는 부분 포함
```

장점:
- 버전 관리된 프롬프트 히스토리
- 에이전트 간 공유 체크리스트 재사용
- 코드 변경 없이 프롬프트 교체
- 순환 참조 감지 및 깊이 제한

## 자동화하지 말아야 할 것

다음은 의도적으로 사람의 판단을 유지합니다:

| 결정 사항 | 이유 |
|----------|------|
| 어떤 슬로우 쿼리를 분석할지 | 지표 너머의 비즈니스 컨텍스트 |
| 제안된 패치를 적용할지 여부 | 기술 + 비즈니스 트레이드오프 |
| 최종 PR 머지 | 책임 소재 |
| 프로덕션 배포 | 비가역적, 높은 영향 범위 |

## 도구 사용

```bash
# 모든 에이전트 하네스 정의 검증
claude-harness check-all

# 단일 에이전트 검증
claude-harness validate agents/09-harness-designer.md

# 타이트 하네스 템플릿 생성
claude-harness template tight my-specialist > agents/11-my-specialist.md

# 파이프라인 실행 추적
claude-pipeline init my-workflow
claude-pipeline stage "분석" start
claude-pipeline stage "분석" pass --note "3개의 문제 발견"
claude-pipeline stage "패치 생성" start
claude-pipeline report
```
